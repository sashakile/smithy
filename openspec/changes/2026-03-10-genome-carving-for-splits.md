# DQ-5: Genome Carving for Component Splits

**Status:** proposed
**Date:** 2026-03-10
**ID:** DQ-5

## Problem

When a component is split into two children (Section 4.10 topology adaptation), how are the
genomes divided?

The parent has `{:in ParentIn :out ParentOut}`. The children need genomes that together cover
the parent's input→output contract without either:
1. Duplicating the full parent genome on each child (violates the decomposition purpose)
2. Losing schema coverage (children must collectively handle all valid parent inputs)

A routing predicate must also be defined to decide which child handles a given request.

## Proposal

**Children share parent genome `:in`. Each gets a carved `:out` sub-schema. Routing predicate
is another Cell.**

- Child A genome: `{:in ParentIn :out ChildAOut}`
- Child B genome: `{:in ParentIn :out ChildBOut}`
- Router Cell genome: `{:in ParentIn :out [:enum :child-a :child-b]}`
  The Router is a first-class Cell with its own genome — its `:out` is a routing decision,
  not the parent's `:out`. The Router's `:out` schema is generated by `mr plan --topology`
  with one enum variant per child.
- Router Cell starts at P4 (LLM decides routing); differentiates to P1 as routing patterns
  stabilize.

The carved `:out` schemas are generated by `mr plan --topology` based on which output fields
each cluster populates (output-shape-sig analysis from Section 4.10.5).

## Alternatives

1. **Carved `:in` schemas** — Children receive only the input fields relevant to their cluster.
   Cleaner separation but requires the routing predicate to understand input structure, and any
   input field used by both children must be duplicated.
2. **Shared full genomes** — No carving; children are identical to parent. Defeats the purpose
   of splitting (they'll just re-converge to the same behavior).

## Decision

Pending. Recommended: shared `:in`, carved `:out`, routing predicate as its own differentiable
Cell.
