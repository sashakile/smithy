## Purpose
Define autonomous topology adaptation: split and fuse detection signals, proposal lifecycle, human review requirements, and component deprecation windows.

## Requirements

### Requirement: Topology adaptation is opt-in and defaults off
The system SHALL require explicit opt-in to topology adaptation via `:topology-adaptation {:enabled true}`
in smithy.edn. The `:auto-commit` key SHALL default to false and the system SHALL emit a
startup warning if `:auto-commit true` is set.

#### Scenario: Topology adaptation is disabled by default
- **WHEN** smithy.edn does not include a `:topology-adaptation` section
- **THEN** no split or fuse proposals are generated regardless of signaling history patterns

#### Scenario: auto-commit true triggers startup warning
- **WHEN** DR starts with `:topology-adaptation {:auto-commit true}`
- **THEN** DR logs a warning: "auto-commit is set to true — topology changes will apply without human review"

### Requirement: Split is proposed when any of four signals exceeds threshold
The system SHALL propose a split when any single signal exceeds its threshold:
(1) bimodal confidence distribution (Hartigan's dip > 0.3),
(2) potency ceiling lock (same potency 30+ days with ≥40pp per-cluster coverage gap),
(3) decision path divergence (reasoning fingerprints partition into non-overlapping groups),
(4) schema polymorphism (output-shape-sig distribution entropy is high).

#### Scenario: Bimodal confidence triggers split proposal
- **WHEN** Hartigan's dip statistic on a component's confidence distribution exceeds 0.3
- **THEN** a split proposal is generated identifying the two confidence clusters

#### Scenario: Potency ceiling lock triggers split proposal
- **WHEN** a component has been at the same potency for 30+ days with a per-cluster coverage gap ≥ 40 percentage points
- **THEN** a split proposal is generated

### Requirement: Fuse is proposed when any of four signals exceeds threshold
The system SHALL propose a fuse when any single signal exceeds its threshold across a
component pair: (1) temporal coupling ≥ 95%, (2) schema subsumption,
(3) redundant decision patterns (high Jaccard similarity), (4) cascade waste (A cascades to P4
>20%; B is consistently sub-ms on A's output).

#### Scenario: Temporal coupling triggers fuse proposal
- **WHEN** component A always fires immediately before B and B is never invoked independently (≥95% coupling)
- **THEN** a fuse proposal is generated for A and B

### Requirement: Topology proposals require human review before commit
The system SHALL require human approval via `mr review --topology` before any topology
change (split or fuse) is committed to the Registry. `:auto-commit false` is enforced
at the application layer.

#### Scenario: Split proposal requires explicit approval
- **WHEN** a split proposal is generated by the topology analyzer
- **THEN** it enters `:status :reviewing` and cannot be committed without `mr review` approval

### Requirement: Fused component starts at highest potency of merged components
The system SHALL assign the fused component's initial Fate.potency to the highest potency
level among the merged components (conservative assignment).

#### Scenario: Fusing P1 and P3 starts at P3
- **WHEN** component A (P1) and component B (P3) are fused
- **THEN** the fused component starts at P3 (the higher of the two)

### Requirement: Deprecated components enter a configurable migration window
The system SHALL keep deprecated components (superseded by split/fuse) active for a
configurable migration window (default: 14 days) before archiving. Signaling history
SHALL be retained for lineage.

#### Scenario: Deprecated component serves requests during migration window
- **WHEN** a component is deprecated by a topology change
- **THEN** it continues to serve requests for the configured migration window period

#### Scenario: Deprecated component is archived after migration window
- **WHEN** the migration window expires
- **THEN** the component enters archived state; signaling history is retained for lineage
